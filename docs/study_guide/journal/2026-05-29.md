## 오늘 배운 핵심
1. **이 NPU의 검증된 수치**: Peak Compute = **51.2 GOPS** (256 MAC × 2 × 100MHz), Peak BW = **3.2 GB/s** (256-bit AXI × 100MHz), **Ridge point = 16 ops/byte**.
2. **Roofline 판정 규칙**: AI < Ridge → memory-bound, AI > Ridge → compute-bound. Ridge는 *기계*의 밸런스 = Peak compute ÷ Peak BW.
3. **GEMM 크기별 판정**: (16³)·(1,2048,2048) = memory-bound, (64³) = 정확히 Ridge, (256³) = compute-bound. → 작은 타일/벡터연산은 PE가 놀고, 64³+ 큰 GEMM이라야 풀가동.
4. **PYNQ-Z2 오버레이의 진짜 병목**: 32-bit 제어경로(3.2 Gbit/s)만 쓰면 Ridge=128 → 거의 모든 workload가 memory-bound. 하드웨어 속도가 아니라 데이터 공급이 문제.

## 오늘 내린 결정
- **결정 1 (중요)**: 학습 가이드 `06_systolic_array.md`의 "Ridge AI ≈ 32, BW 1.6 GB/s"와 실제 `roofline.py`의 "Ridge=16, BW 3.2 GB/s" **불일치를 발견** → 일기에 기록하고 진짜 기준은 코드(off-chip BW)로 잡는다.
  - **이유**: 가이드의 32는 *어레이 입구* 스트리밍 강도(2N, activation만 1바이트)이고, roofline.py의 16은 *외부 DRAM* 밸런스(act+wt+out, 2/2/4바이트)다. "memory-bound 판정"에 쓸 진짜 Ridge는 off-chip 기준 = 16.
  - **영향**: 앞으로 roofline 해석은 코드 기준(Ridge=16) 사용. 가이드의 2N=32는 "어레이 내부 특성치"로만 인용. → 추후 가이드 문구 보강 후보.
- **결정 2**: W1 산출물을 일기 + `roofline_summary.md` 1쪽으로 확정. 로드맵의 W4(roofline.py 코드 정독)·W5(matplotlib 그래프, 실습 1)은 다음 세션 이후로.

## 막힌 부분 / 의문
- Q1: 가이드의 BW 정의(1.6 GB/s 단방향 어레이 입구) vs 코드의 BW(3.2 GB/s 외부 AXI) — 어느 쪽이 "교과서적으로 옳은" roofline 축인가? → off-chip BW가 표준이지만, on-chip SRAM 대역폭 기준 roofline도 따로 그릴 가치가 있는지 확인 필요 (H&P 4.8 정독 시).
- Q2: `roofline.py`의 lane slot이 INT8인데 byte 계산은 2바이트(16-bit slot)로 함. 실제 INT8 payload는 1바이트인데 slot이 16-bit라 2바이트로 센 것 → 실 DRAM 전송량과의 괴리 여부 확인 (emitter/DMA가 16-bit slot 그대로 전송하는지).
