# 28. `ir.py` — Mini IR (Intermediate Representation)

> **파일 위치**: `l6/src/l6_toolchain/ir.py` (848줄)
> **역할**: PyTorch 모델을 추상화한 **그래프 표현**. 12개 op 지원 + validation + numpy 실행기 + 패키지 export
> **선수 지식**: [27_common.md](27_common.md), `dataclasses`, numpy 기본
> **공장 비유**: 본사 설계도면 — 각 부품(op) 사이즈/입출력/연결 관계를 명확히 기록한 표준 양식

---

## 이 파일이 중요한 이유

L6 컴파일러의 **중심 자료구조**입니다. tracer가 이걸 만들고, scheduler가 이걸 읽고, emitter가 이걸 hardware package로 변환합니다. 이 파일을 이해하면 L6 전체 데이터 플로우의 80%를 이해한 셈입니다.

---

## 1. 자료구조 (L14-64)

### `TensorValue` (L14-18)

```python
@dataclass(frozen=True)
class TensorValue:
    name: str
    shape: tuple[int, ...]
    dtype: str = "int8"
```

**무엇**: 그래프 안의 텐서 1개를 표현. 이름 + shape + dtype.

**왜 frozen**: hashable해서 dict의 key로 쓸 수 있고, 실수로 수정될 일이 없음.

**dtype default가 "int8"**: NPU 기본 데이터타입이 INT8이라.

### `OpNode` (L21-27)

```python
@dataclass(frozen=True)
class OpNode:
    name: str
    kind: str           # "linear", "conv2d", "relu", ...
    inputs: tuple[str, ...]   # 입력 tensor 이름들
    outputs: tuple[str, ...]  # 출력 tensor 이름들
    attrs: dict[str, Any] = field(default_factory=dict)
```

**무엇**: 연산 노드 1개. 12종 op (`linear`, `conv2d`, `relu`, `add`, `batch_norm`, `max_pool2d`, `adaptive_avg_pool2d`, `flatten`, `avg_pool2d`, `sigmoid`, `gelu`, `mul`).

**왜 inputs/outputs가 이름 튜플**: 그래프 연결을 **이름**으로 표현 → 그래프 직렬화/역직렬화가 쉬움.

### `Program` (L30-38)

```python
@dataclass(frozen=True)
class Program:
    inputs: tuple[TensorValue, ...]   # 그래프 입력
    tensors: tuple[TensorValue, ...]  # 모든 텐서 (입력+중간+출력)
    ops: tuple[OpNode, ...]           # 연산 시퀀스
    outputs: tuple[str, ...]          # 그래프 출력 tensor 이름들

    def tensor_map(self) -> dict[str, TensorValue]:
        return {tensor.name: tensor for tensor in self.tensors}
```

**무엇**: 전체 모델. inputs ⊆ tensors. ops는 topological order로 가정.

## 2. Validation (L66-99, L388-509)

### `validate_program` (L66-99)

각 op에 대해:
1. inputs/outputs가 비어있지 않은지
2. 모든 input tensor가 `program.tensors`에 등록되어 있는지
3. 모든 input tensor가 **이전 op의 output 또는 program input**인지 (use-before-def 방지)
4. 모든 output tensor가 다른 곳에서 안 쓰이는지 (SSA 보장)
5. op 별 세부 형상 검사 (`_validate_op_node`)

```python
seen_outputs: set[str] = set(tensor.name for tensor in program.inputs)
for op in program.ops:
    for input_name in op.inputs:
        if input_name not in seen_outputs:
            raise ValueError(f"Op {op.name} consumes tensor {input_name} before it is produced.")
    for output_name in op.outputs:
        if output_name in seen_outputs:
            raise ValueError(f"Tensor {output_name} is produced more than once.")
        seen_outputs.add(output_name)
```

### `_validate_op_node` op-별 검사 (L388-509)

각 op kind마다 형상 규칙이 다릅니다. 예를 들어 `linear`:

```python
if op.kind == "linear":
    if len(op.inputs) != 1 or len(op.outputs) != 1:
        raise ValueError(f"Linear op {op.name} must have exactly one input and one output.")
    input_tensor = tensor_map[op.inputs[0]]
    output_tensor = tensor_map[op.outputs[0]]
    if len(input_tensor.shape) != 2 or len(output_tensor.shape) != 2:
        raise ValueError(f"Linear op {op.name} expects rank-2 input and output tensors.")
    in_features = _require_attr_int(op, "in_features")
    ...
    if output_tensor.shape != (input_tensor.shape[0], out_features):
        raise ValueError(f"Linear op {op.name} output shape must be (batch, out_features).")
```

**핵심 패턴**: **shape 검사 + attr 검사 + dimension matching**. 모든 op가 같은 패턴.

| `conv2d` | L404-426 |

## 3. Lowering (L102-318)

### `lower_program_to_steps` (L102-318)

**역할**: `Program` (사용자가 표현한 op) → `list[LoweredOp]` (백엔드가 직접 처리할 형태).

**왜 lowering이 필요한가**:
- `linear` → `gemm` (행렬곱으로 분해)
- `conv2d` → `conv2d_im2col_gemm` (im2col로 펼친 행렬곱)
- `relu`, `add`, `batch_norm`, sigmoid 등 → `elementwise_post_op` (bytecode 측면에서 동일 카테고리)
- `flatten` → `shape_only` (계산 없음, 단순 reshape)

**3가지 lowered_kind 카테고리**:
| 카테고리 | 의미 | 예시 |
|---|---|---|
| `gemm` | systolic array 1회 발사 | linear |
| `conv2d_im2col_gemm` | im2col로 펼친 후 systolic | conv2d |
| `elementwise_post_op` | scalar 연산, 별도 백엔드 처리 | relu, add, sigmoid, gelu, ... |
| `shape_only` | 데이터 이동 없음 | flatten |

**예시: linear → gemm 매핑** (L108-129):

```python
if op.kind == "linear":
    input_tensor = tensor_map[op.inputs[0]]
    in_features = _require_attr_int(op, "in_features")
    out_features = _require_attr_int(op, "out_features")
    lowered_steps.append(LoweredOp(
        name=op.name,
        source_kind=op.kind,
        lowered_kind="gemm",
        inputs=op.inputs,
        outputs=op.outputs,
        attrs={
            "m": input_tensor.shape[0],   # batch
            "k": in_features,
            "n": out_features,
            "logical_input_shape": input_tensor.shape,
            "logical_output_shape": output_tensor.shape,
        },
    ))
```

**핵심**: `(m, k, n)` triple로 변환 → systolic array가 직접 이해할 수 있는 형태.

## 4. Export + Runtime Execution (L321-848)

### `export_program_package` (L321-385)

**역할**: `Program` → 디스크에 workload package 디렉토리 작성.

**산출물**:
```
output_dir/
├── manifest.json           # 패키지 메타데이터
├── steps.json              # step 인덱스
├── README.md               # 자동 생성 설명
└── steps/
    ├── step_000_<name>/
    │   ├── manifest.json   # 이 step의 메타데이터
    │   └── compute_package/   # gemm/conv 한정 — emitter.py로 생성
    └── step_001_<name>/
        └── ...
```

**3가지 동작 모드**:
1. `tensor_data=None`: shape만 가지고 패키지 생성 (스켈레톤)
2. `tensor_data={x: ...}`: 입력 텐서 실제 값을 받고 step별 runtime tensor 계산
3. `schedule_metadata=...`: scheduler가 미리 정한 tile 순서/사이클 정보를 step manifest에 기록

### `_execute_program_op` (L537-694)

**역할**: 한 op를 실제 numpy로 실행 → runtime tensor를 갱신.

**왜 numpy로 실행**:
1. **Golden generation**: hardware가 계산할 결과를 미리 계산해서 `golden.bin`으로 저장
2. **Multi-step chain**: 다음 step의 입력은 이전 step의 출력 → 자동으로 연결
3. **Replay validation**: hardware 결과와 numpy 결과를 비교

**예시: linear 실행** (L540-556):

```python
if op.kind == "linear":
    in_features = _require_attr_int(op, "in_features")
    out_features = _require_attr_int(op, "out_features")
    weight = _to_numpy_array(_require_attr_value(op, "weight"))
    bias = _optional_bias(op, expected_width=out_features)

    linear_data = LinearData(activations=input_array, weights=weight.T, bias=bias)
    output = linear_data.activations.astype(np.int32) @ linear_data.weights.astype(np.int32)
    if bias is not None:
        output = output + bias.astype(np.int32)
    runtime_tensors[op.outputs[0]] = output.astype(np.int32)
    return linear_data
```

**핵심 디테일**:
- `astype(np.int32)`: INT8 입력을 INT32로 promote → 곱셈 overflow 방지 (NPU 하드웨어와 동일)
- `weight.T`: PyTorch는 `(out, in)`, GEMM은 `(in, out)` → transpose
- `LinearData` 반환: emitter가 골든을 만들 때 그대로 사용

**12개 op execution 위치**:
| op | 라인 | 백엔드 |
|---|---|---|
| `linear` | L540-556 | numpy `@` (INT32 promote) |
| `conv2d` | L558-607 | im2col + GEMM |
| `relu` | L609-611 | `np.maximum(x, 0)` |
| `add` | L613-615 | INT32 promote 후 합 |
| `batch_norm` | L618-630 | float64 계산 후 INT32 cast |
| `max_pool2d` | L632-649 | 명시적 sliding window loop |
| `adaptive_avg_pool2d` | L651-664 | adaptive index 계산 |
| `flatten` | L666-671 | `reshape` |
| `avg_pool2d` | L673-691 | sum // window_size |
| `sigmoid` | L693-697 | `1/(1+exp(-x)) * 127` (INT8 range mapping) |
| `gelu` | L699-705 | tanh-approx |
| `mul` | L707-709 | INT32 promote 후 곱 |

## 핵심 개념 정리

| 개념 | 의미 | 어디서 |
|---|---|---|
| **SSA (Single Static Assignment)** | 각 텐서는 정확히 한 번 정의됨 | `validate_program` |
| **Op Lowering** | 사용자 op → 백엔드 가능 카테고리 | `lower_program_to_steps` |
| **Runtime Execution** | numpy로 동일 산식 실행 → golden 자동 생성 | `_execute_program_op` |
| **Multi-step Chain** | step i 출력 = step i+1 입력 (자동 전파) | `runtime_tensors` dict |
| **INT8 → INT32 promotion** | 곱셈 결과 overflow 방지 | linear/conv/add/mul executor |

## 연습 문제

1. `Program(ops=(linear, relu, linear))`에서 두 번째 `linear`의 입력은 어디서 오는가? `validate_program`이 어떻게 검사하는가?
2. `lowered_kind`가 4가지(`gemm`, `conv2d_im2col_gemm`, `elementwise_post_op`, `shape_only`)인 이유는? 백엔드 관점에서 왜 이렇게 분류했을까?
3. `_execute_program_op`이 INT8 입력에 `astype(np.int32)`를 거는 이유는? 만약 안 걸면 어떤 결과가 나올까?
4. (실습) `relu` 다음에 `mul` op를 추가하고 `validate_program`이 통과하는지 확인. 두 입력 shape가 다르면 어떤 에러?
