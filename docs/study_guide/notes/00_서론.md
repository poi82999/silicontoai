# 📓 필기 노트 — 서론: 왜 나는 이 노트를 쓰는가

> 시작: 2026-07-23 · 이 폴더(`notes/`)는 **백본 노드 1→6을 정면돌파하며 내가 직접 쓰는 학습 필기**다.
> 성격: 레퍼런스(`../roofline_notes.md`)나 코드 워크스루(`../08_l6_toolchain/`)와 다르다. 여기는 **유도 과정과 내가 막힌 지점**을 1인칭으로 남기는 곳.

---

## 0. 나의 좌표

나는 KAIST 전산학부 3학년이고, 16×16 Weight-Stationary Systolic Array NPU를 **RTL → Python 컴파일러(l6) → UVM → FPGA까지 풀스택으로 이미 완주**했다. 그래서 이 여정은 "새로 배우는" 게 아니라 **내가 만든 것을 이론의 언어로 다시 해석하는** 작업이다.

목표(Track A+): **HW를 아는 ML 컴파일러/아키텍트.** 순수 VLSI가 아니라 HW↔SW 두 멘탈 모델을 동시에 쥐는 co-design. 그게 AI에 대체되지 않는 가장 희귀한 자리라고 판단했다.

## 1. 왜 굳이 노트를 쓰는가 — 학습감은 경계에서 나온다

지난 7주 공백에서 배운 것: **"공부하는 느낌이 안 든다"는 정확한 신호였다.** 이미 아는 것(자료구조·알고리즘)을 다시 주행하면 마찰이 없고, 마찰이 없으면 학습도 없다. 학습감은 **지식의 경계에 설 때만** 생긴다.

노트는 그 경계를 인위적으로 계속 만들어 주는 장치다. "안다"고 착각하던 것을 글로 쓰려는 순간 경계가 드러난다 — Ridge point를 손으로 유도하려다 `×2`가 왜 붙는지 막히는 그 순간이 실제 학습이 일어나는 지점이다. **막힌 곳을 지우지 않고 남기는 것**이 이 노트의 핵심 규칙이다.

## 2. 노트 작성 원칙 — 한 개념을 닫는 5단계

모든 노드에 똑같이 적용하는 틀. 이 순서로 써야 "안다"가 착각이 아니게 된다.

```
① 이론 골격  — 논문/교과서 1편으로 핵심 공식·용어만 (암기 X)
② 내 시스템 대입 — 내 NPU의 실제 숫자로 ①을 직접 재유도
③ 실물 검증  — 코드 돌려 숫자 확인 (여기서 문서 버그도 잡힌다)
④ 쟁점 결론  — 내가 헷갈린 것을 스스로 결론낼 때까지 파기
⑤ 산출물 고정 — 그래프·1쪽 문서로 박제 = "졸업 증거"
```

**졸업 판정**: 해당 노드의 이해도 점검 질문을 *안 보고* 즉답할 수 있으면 다음 노드로. 못 하면 ④가 안 닫힌 것.

## 3. 두 가지 레일 규칙 (로드맵에서 가져옴)

- **백본은 push** — 노드 1→6을 순서대로 정면돌파. 곁가지로 새지 않는다.
- **기초는 pull** — 이미 배운 기초(CS206/CS300 등)는 독립 트랙으로 재학습 금지. 백본을 파다 필요해진 개념만 그 자리에서 10~30분 복습 후 복귀. 복습이 생기면 일기에 `pull: 개념명` 한 줄.
- **KAIST 중복은 스킵** — 학교가 다룰 것(CS311 컴퓨터구조, CS420 컴파일러)은 학교에 맡기고, 노트는 학교가 안 다루는 NPU 고유 가치에만 쓴다.

## 5. 왜 roofline부터인가

성능 모델은 **가속기 세계의 공용어**다. "이 커널이 계산에 막혔나, 데이터 공급에 막혔나(compute-bound / memory-bound)"라는 한 문장 없이는 노드 2~6의 어떤 결정도 논할 수 없다 — 어레이를 왜 그 크기로 했나(노드 3), scheduler가 왜 그 타일 순서를 고르나(노드 5)가 전부 이 어휘 위에서 굴러간다.

그래서 노트의 첫 장은 roofline이고, 그게 [01_roofline.md](01_roofline.md)다.
