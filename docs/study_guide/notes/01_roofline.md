## §0. 문제의식 — 나는 왜 이게 필요했나

내 NPU를 다 만들었을 때 답할 수 없던 질문이 있었다: **"이거 빠른 건가? 느리면 어디가 문제인가?"**
칩을 돌려 51.2 GOPS가 나온다 쳐도, 그게 잘 나온 건지 못 나온 건지 판단할 **자(尺)**가 없었다.

Roofline의 출발점은 무섭도록 단순한 관찰이다:

> 💡 가속기가 느린 이유는 **딱 둘 중 하나**다. ① 계산기(PE)가 부족하거나, ② 데이터를 충분히 빨리 못 갖다주거나. 세 번째는 없다.

이 둘을 한 그림에 그려서 "지금 나는 어느 쪽에 막혀 있나"를 보여주는 게 roofline이다.

## §1. Williams의 아이디어 (2009)

논문 한 줄: *어떤 커널의 성능은 두 천장 — 연산 능력과 메모리 대역폭 — 중 낮은 쪽에 막힌다.* 이걸 **operational intensity** 축 위의 로그-로그 그래프로 그린다.

### Operational Intensity (OI)
```
OI = (총 연산 수) / (DRAM으로 오간 총 바이트)     [ops / byte]
```
> ❓ 왜 하필 *DRAM 트래픽* 기준인가? → 캐시/SRAM에서 재사용을 잘 하면 DRAM으로 나가는 바이트가 줄어든다. 그러면 같은 연산을 더 적은 메모리 트래픽으로 해내니 OI가 올라간다. 즉 **OI는 "데이터 재사용을 얼마나 잘 하는가"를 한 숫자로 압축한 것.** 이게 이 모델의 진짜 통찰이다.

### 성능 공식 (유도)
계산 파이프라인과 메모리 파이프라인이 병렬로 돈다고 보면, 실제 성능은 둘 중 느린 쪽이 정한다:
```
달성 성능 = min( Peak Compute ,  BW × OI )
                    └ 수평 천장      └ 대각선 (기울기 = BW)
```
- OI가 작으면(재사용 적음) → `BW × OI`가 작아 대각선에 먼저 막힌다 = **memory-bound**
- OI가 크면(재사용 많음) → `Peak Compute`에 막힌다 = **compute-bound**

### 손으로 그린 그래프
```
성능(GOPS,log) ▲
               │              ┌─────────── Peak Compute  (기계가 못 넘는 수평선)
               │             ╱
               │            ╱   기울기 = BW
               │           ╱
               │          ● ← Ridge point (두 천장이 만남)
               │         ╱
               │        ╱
               └───────┴──────────────────► OI (ops/byte, log)
                 memory-bound │ compute-bound
```

### 💡 Ceilings — 천장은 한 겹이 아니다
Peak는 최적화를 다 했을 때만 나온다. 논문은 천장을 **여러 겹**으로 그린다:
- 연산 쪽: SIMD 안 쓰면 / ILP 부족하면 / mul·add 불균형이면 → 더 낮은 수평선들
- 메모리 쪽: prefetch 없으면 / NUMA 최적화 없으면 → 더 낮은 대각선들

이 겹겹의 천장이 곧 **"어떤 순서로 최적화하라"는 로드맵**이다. 그냥 성능 예측 도구가 아니라 *처방* 도구라는 게 핵심.
🔗 연산 천장의 "SIMD 겹"은 노드 1의 다음 주제(SIMD/DLP)에서, 메모리 계층은 노드 2(CS311)에서 다시 만난다.

## §2. Ridge Point — 이 노트의 심장

두 천장이 만나는 지점의 OI. 유도는 그냥 두 식을 같다고 놓는 것:
```
Peak Compute = BW × OI*   →   OI* = Peak Compute / BW      [ops/byte]
```

> 💡 직관: Ridge는 **"이 기계에서 peak를 뽑으려면 최소한 바이트당 몇 번은 계산해야 하는가"**의 문턱값이다.
> - Ridge가 **왼쪽**(작음)에 있으면 → 웬만한 커널도 쉽게 compute-bound = peak 뽑기 쉬운 기계
> - Ridge가 **오른쪽**(큼)에 있으면 → 어지간히 재사용해도 memory-bound = peak 뽑기 어려운 기계

이 한 숫자가 기계의 성격을 규정한다. 그래서 다음 절에서 내 NPU의 Ridge를 손으로 구하는 게 이 장의 절정이다.

---

## §3. 내 NPU에 손으로 대입 (sim_default: 16×16, INT8, 100MHz, 256-bit AXI)

### ① Peak Compute
```
Peak Compute = (MAC 수) × (MAC당 ops) × (clock)
             = 256      × 2           × 100 MHz
             = 256 × 2 × 100×10⁶
             = 51.2 × 10⁹ ops/s = 51.2 GOPS
```
> ❓ 왜 `×2`? → 1 MAC = 곱 1번 + 합 1번 = **2 ops**. ops를 셀 때 `2MKN`으로 세니까, peak에도 반드시 `×2`가 붙어야 단위가 맞는다.
> ⚠️ **함정**: 어떤 문서엔 Peak가 `256×100/1000 = 25.6 GOPS`로 적혀 있다(`×2` 누락). 실제 코드(`roofline.py` L98)·매니페스트는 51.2가 맞다. 이건 내가 정리하다 잡은 실제 버그다. → [../roofline_notes.md §8](../roofline_notes.md)

### ② Peak BW (외부 AXI)
```
Peak BW = (버스 폭) × (clock)
        = 256 bit × 100 MHz = 25.6 × 10⁹ bit/s = 25.6 Gbit/s
        = 25.6 / 8 = 3.2 GB/s
```
> ⚠️ 25.6은 Gbit/s, 3.2는 GB/s — **같은 양, 다른 단위.** 코드는 Gbit/s로 입력받아 내부에서 ÷8 한다. 혼동 주의.

### ③ Ridge Point ★
```
Ridge = Peak Compute / Peak BW = 51.2 GOPS / 3.2 GB/s = 16 ops/byte
```

> ┌──────────────────────────────────────────────┐
> │  내 NPU:  Peak 51.2 GOPS · BW 3.2 GB/s · **Ridge = 16**  │
> └──────────────────────────────────────────────┘
>
> 💡 해석: 내 칩에서 PE를 풀가동시키려면 **DRAM 바이트 하나당 최소 16번은 계산**해야 한다. 그보다 재사용이 적은 커널은 무조건 메모리에 굶는다.

## §4. GEMM 크기별 — 직접 판정해보기

GEMM (M,K,N)에 대해:
```
ops   = 2·M·K·N
bytes = M·K·(2)  +  K·N·(2)  +  M·N·(4)      (act 2B, weight 2B, out 4B)
        └입력      └가중치       └출력(누산기)
OI    = ops / bytes
```

네 케이스를 손으로:

| (M,K,N) | ops | bytes | OI | vs Ridge(16) | 판정 |
|---|---|---|---|---|---|
| (16,16,16) | 8,192 | 512+512+1024=2,048 | **4** | < 16 | memory-bound (util 25%) |
| (64,64,64) | 524,288 | 8,192·2+16,384=32,768 | **16** | = 16 | **정확히 Ridge** (util 100%) |
| (256,256,256) | 33.5M | 524,288 | **64** | > 16 | compute-bound (풀가동) |
| (1,2048,2048) | 8.39M | ~8.40M | **≈1** | ≪ 16 | memory-bound 최악 (util 6%) |

> 💡 결론: **64³쯤 되는 정방 GEMM이라야 PE가 100% 돈다.** 작은 타일이나 벡터-행렬(M=1)은 OI가 낮아 계산기가 놀고 메모리만 헐떡인다.
> 🔗 (1,2048,2048)이 왜 최악인가 → M=1이면 weight를 한 번씩밖에 못 쓰고 버린다(재사용 0). 이게 §8의 LLM 디코드 문제와 정확히 같은 얼굴이다.

---

## §5. 같은 커널, 다른 기계 — 병목은 이동한다

같은 GEMM이라도 기계가 바뀌면 판정이 뒤집힌다. Ridge = Peak/BW 이니까, **BW를 줄이면 Ridge가 오른쪽으로 밀린다.**

PYNQ-Z2 오버레이: PS7의 32-bit control-plane(`M_AXI_GP0`)으로 붙어서 BW가 **3.2 Gbit/s** — sim보다 8배 작다.
```
Ridge(PYNQ) = 51.2 GOPS / (3.2/8 GB/s) = 51.2 / 0.4 = 128 ops/byte
```
> 💡 Ridge가 16→128로 밀리면, §4의 GEMM들이 **거의 전부 memory-bound**로 떨어진다. 하드웨어 연산속도(51.2 GOPS)는 그대로인데, 데이터 공급이 병목이라 못 써먹는 것. **"빠른 계산기 ≠ 빠른 시스템"**의 교과서적 사례.

## §6. 그래서 뭘 해야 하나 — 처방은 병목마다 다르다

| 병목 | 뭘 하면 빨라지나 | 헛수고 |
|---|---|---|
| **compute-bound** (OI > Ridge) | 클럭↑ · PE 수↑ · 더 좁은 수치형(INT8/FP8) | OI 더 높여봤자 무효 |
| **memory-bound** (OI < Ridge) | **OI 높이기**: 타일링 · fusion · 재사용 · 패킹 / BW↑ | 클럭·PE 추가는 무효 |

> ❓ 자문: *"OI=8 커널을 클럭 2배로 올리면 빨라지나?"*
> → sim_default Ridge=16이라 OI=8은 memory-bound. **클럭 올려도 무효** — 병목은 메모리다. OI를 16 이상으로 끌어올려야(타일링·fusion) 비로소 클럭 상승이 성능으로 전환된다.
> ✅ 이 자문에 즉답이 되면 §6은 닫힌 것.

## §7. 내가 헷갈렸던 두 지점 (여기가 진짜 학습)

### ❓ Q1. 메모리 축의 BW는 16이 맞나 32가 맞나?

내 학습 가이드(`06_systolic_array.md`)는 "Ridge ≈ 2N = 32, BW 1.6 GB/s"라 적혀 있고, 코드는 "Ridge = 16, BW 3.2 GB/s"다. 처음엔 둘 중 하나가 틀린 줄 알았는데, **다른 양을 재고 있는 것**이었다:

| | 가이드 (32) | 코드 (16) |
|---|---|---|
| BW 대상 | 어레이 *입구* 1.6 GB/s | 외부 AXI 3.2 GB/s |
| 세는 bytes | activation만, 1B, 단방향 | act+weight+out, 2/2/4B |
| 의미 | 어레이 **내부** 스트리밍 특성치 | **기계 전체**의 off-chip 밸런스 |

> ✅ 결론: 고전 Roofline(Williams)의 메모리 천장 = **off-chip DRAM 대역폭**이다. 모델의 목적이 "칩 *밖*에서 데이터를 얼마나 빨리 끌어오나"의 한계를 그리는 것이므로. **판정에는 항상 Ridge=16(off-chip)을 쓴다.** 어레이 입구 2N=32는 "내부 특성치"로만 인용.
> 🔗 남은 갈래: *Hierarchical Roofline*(DRAM/SRAM 천장을 겹으로 그림)을 쓰면 둘 다 한 그림에 담긴다 → H&P 4.8절에서 확정 예정.

### ❓ Q2. INT8 연산인데 왜 byte를 2로 세나?

코드가 `input/weight_elem_bytes=2`로 센다. INT8은 1바이트인데?
근거: `npu_def_pkg.sv` — *"Lane slot 폭을 16-bit로 유지, 하위 8bit만 INT8 payload."* 즉 실제 AXI/SRAM은 16-bit slot을 통째로 옮기고 **상위 8bit는 0 패딩.**

> ✅ 결론: 2바이트 카운팅은 *실제 전송량*엔 충실하다(패딩까지 나르니까). 하지만 이건 **설계 낭비** — act/weight 대역폭의 **절반이 0을 나르는 데** 쓰인다. INT8을 1바이트로 패킹하면 OI가 오른다 — 정사각 GEMM은 8/6배(≈1.33×), weight-지배(M=1 디코드류)는 ≈2× (INT32 출력은 패킹 안 되므로 shape 의존). → [실습 1 분석](../../roofline_analysis.md)
> 🔗 이게 노드 6(양자화·INT8/FP8 패킹)으로 이어지는 실마리다. "l6에 패킹 최적화 추가"가 잠재 프로젝트.

## §8. 세상 속 좌표 + 2026 렌즈

| 설계 | 어레이 | full-GEMM OI | 성격 |
|---|---|---|---|
| 내 NPU | 16×16 | ~32(2N) / off-chip 16 | FPGA mid-range, BRAM 대역폭에 맞춤 |
| TPU v1 | 256×256 | ≈512 | 극단적 compute-bound, DRAM 충분한 데이터센터용 |
| Eyeriss | Row-Stationary | — | 에너지·재사용 최적화(dataflow 축) |

> 💡 **2026 최전선 = memory wall.** LLM 디코드는 토큰당 M=1 벡터-행렬이라 §4의 (1,2048,2048)처럼 **OI≈1의 깊은 memory-bound**. attention 커널 사이클의 절반 이상이 메모리 대기다. CNN 시대에 그린 "큰 GEMM으로 compute-bound 달성" 그림이 LLM 추론에선 무너진다.
> 🔗 노드 3에서 TPU/Eyeriss 논문 읽을 때 각 논문마다 **"이 설계가 decode에서 무너지는 지점" 한 문단**을 붙일 것.

## §9. 스스로 점검 (안 보고 답하기)

1. **클럭을 200MHz로 올리면 Ridge는?** → Peak Compute·Peak BW가 *둘 다* 2배 → Ridge = 102.4/6.4 = **16, 불변.** (기계 밸런스는 클럭에 안 변한다.)
2. **OI=8 커널은 클럭 2배로 빨라지나?** → No. Ridge=16 미만이라 memory-bound. §6.
3. **INT8 1바이트 패킹하면 (64,64,64)는 여전히 Ridge에 걸리나?** → 정사각 GEMM은 8/6배(≈1.33×)만 → OI 16→**21.3** → compute-bound로 **넘어간다**(간신히). ※"2배→32"는 틀림 — INT32 출력은 패킹 안 되므로. → [실습 1 분석](../../roofline_analysis.md)
4. **(1,2048,2048)이 대각선에 붙는 이유 한 문장?** → M=1이라 weight 재사용이 0, OI≈1로 극단적 memory-bound라서.

> ✅ 이 4개를 안 보고 즉답하면 → roofline 개념부는 졸업. 남은 건 산출물(실습 1)뿐.
